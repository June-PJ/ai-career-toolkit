# Interview Griller（模拟面试打磨器） / Mock Interview Drill

## Overview

多模式模拟面试工具，基于用户简历逐项目深挖追问，同时支持行为面试、系统设计面试和全流程模拟。核心场景：简历经过包装后，验证自己能否在面试中兜住。纯对话驱动，无外部依赖，支持中/英双语面试。

## When to Use

- 简历写完想验证能不能兜住
- 面试前想模拟被拷打
- 想知道简历哪些点容易被问穿
- `resume-builder` 输出后想自测充水部分
- 准备行为面试（STAR）、系统设计面试
- 模拟完整面试流程（一面 → 二面 → 三面 → HR面）

## 面试模式

### 一面（基础追问 / Technical Phone Screen）

围绕简历项目中的技术栈，考察"是什么/怎么做的/为什么这样做"，模拟第一轮技术面。

- 从项目描述的技术点出发提问
- 答对 → 换角度再追一层（"那并发场景呢？"）
- 答错 → 给一句提示（"想想 CAP 定理"），记录薄弱点，继续
- 完全不会 → 给答案框架（"一般从 X、Y、Z 三个角度回答"），标记盲区，下一题
- 可追问相关八股文（限定在简历技术栈范围内）
- 职级适配：Junior 侧重基础知识掌握，Senior 侧重原理深度

### 二面（架构决策深挖 / Architecture Deep Dive）

围绕简历项目的架构选型和设计决策，模拟第二轮深度技术面。

- 问"为什么选这个方案？有什么替代方案？边界场景怎么处理"
- 重点：技术选型理由、容灾方案、可扩展性、失败场景
- 故意提出挑战性假设（"数据量翻 100 倍怎么办？""这个中间件挂了呢？"）
- 考察 trade-off 意识和决策表达能力

### 三面 / 交叉面（系统设计 + 跨团队协作 / Cross-team Design）

模拟第三轮或交叉面，侧重系统设计能力和跨团队协作场景。

- **系统设计题**：面试官根据用户简历的技术栈出设计题（如"设计一个XX系统"）
  - 考察需求澄清、架构选型、数据结构、可扩展性、异常处理
  - 提供白板式引导（"先说说需求范围"、"考虑一下数据模型"）
- **跨团队协作场景**：
  - "如果这个接口需要依赖另一个团队，对方排期跟不上怎么办？"
  - "技术方案评审时被其他团队挑战，怎么应对？"
  - "线上事故定责，怎么处理？"
- **Code Review 场景**：
  - 描述一段伪代码场景，让用户指出问题并给出改进建议

### HR面（行为面试 / Behavioral Interview - STAR Method）

基于简历经历进行行为面试，使用 STAR 方法评估。

- **提问范围**：
  - 领导力/主动性："举一个你主动推动改进的例子"
  - 冲突处理："和同事有技术分歧怎么解决"
  - 失败经历："讲一个你搞砸了的项目"
  - 团队协作："在跨部门项目中你的角色"
  - 成长动机："为什么离开上一家公司"
  - 职业规划："接下来三年的规划"
- 要求用户按 STAR 结构回答
- 针对每个要素（Situation / Task / Action / Result）评估完整性
- 提示缺失的要素并给出改进建议

### 全流程模拟（Full Loop Simulation）

按真实面试流程依次进行多轮模拟：

1. 一面（技术基础）— 30-40分钟
2. 二面（架构深度）— 30-40分钟
3. 三面/交叉面（系统设计 + 跨团队协作）— 40-50分钟
4. HR面（行为面试）— 20-30分钟

每轮结束给出简短口头反馈，全流程结束后输出综合报告。

## 面试官风格

### 友好引导
- 追问温和，循序渐进
- 答不上来给更多提示和引导
- 鼓励式反馈（"方向对了，再想想还有什么？"）

### 压力追问
- 连环比问，不给喘息
- 故意挑衅（"这个说法有漏洞"、"你确定？"）
- 评分客观公正，不因风格影响打分

## 对话行为规则

- 一次只问一个问题
- 不提前暴露答案（除非用户明确放弃该题）
- 追问链最多 3-4 层，不死磕一个点
- 每切换项目时告知进度
- 答不上来当场给简短提示 + 答案方向，不展开完整讲解
- 用户随时可喊停

## 自适应难度

| 上一题等级 | 调整策略 |
|-----------|---------|
| A | 下一题增加追问深度，提出更具挑战性的假设 |
| B | 保持当前难度，再多问一层细节 |
| C | 降低追问深度，增加引导性提示 |
| D | 先给解析，下一个不同方向的问题降低难度 |

# 学习报告

## 📊 薄弱点全景

| 薄弱点 | 等级 | 紧急度 | 涉及项目 |
|--------|------|--------|---------|
| Kafka vs RabbitMQ 选型 | D | 🔴 高 | XXX系统 |

## 🔴 D 级盲区 — 完全不会

### 1. Kafka vs RabbitMQ 选型对比
- **面试场景**："为什么选 Kafka 而不是 RabbitMQ？"
- **正确回答框架**：
  ```
  从三个方面对比：
  1. 消息模型：Kafka 分区日志 vs RabbitMQ 队列
  2. 适用场景：Kafka 适合高吞吐流式 → 日志/指标；RMQ 适合可靠投递 → 任务调度
  3. 关键差异：消息回溯、顺序消费、Exactly-Once 语义
  ```
- **实战例子对比**：
  | 场景 | 推荐 | 原因 |
  |------|------|------|
  | 订单消息通知 | RabbitMQ | 需要可靠投递、死信队列 |
  | 用户行为日志 | Kafka | 高吞吐、消息回溯 |
  | 实时推荐计算 | Kafka | 流式计算集成 |
  | 定时任务调度 | RabbitMQ | 延迟队列支持 |
- **推荐学习**：
  1. 📖 《Kafka 权威指南》第 1-3 章
  2. 📖 《RabbitMQ 实战》第 2 章
  3. 🔗 对比文章：search "Kafka vs RabbitMQ 选型对比"

### 2. 分布式事务
- **面试场景**："订单系统和库存系统怎么保证一致性？"
- **正确回答框架**：
  ```
  常见方案：
  1. 两阶段提交（2PC）— 强一致但性能差
  2. TCC（Try-Confirm-Cancel）— 业务侵入大
  3. 本地消息表 + 最终一致 — 最常用
  4. Seata AT 模式 — 框架自动化
  ```
- **实战例子**：
  ```java
  // 本地消息表方案
  @Transactional
  public void createOrder(Order order) {
      // 1. 插入订单
      orderDao.insert(order);
      // 2. 插入本地消息表（状态=待发送）
      messageDao.insert(new Message("order.created", order));
      // 3. job 轮询发送消息到 MQ，消费端处理库存扣减
  }
  ```

## 🟡 C 级薄弱点 — 知道但说不清

### 1. 缓存一致性
- **你的回答**：提到 Redis 做缓存，但没说明更新策略
- **完整方案**：
  ```
  写策略对比：
  - Cache-Aside：读 miss 回源，写删缓存（最推荐）
  - Read/Write-Through：缓存代理层
  - Write-Behind：异步写回 DB
  一致性问题：删缓存失败 → 重试机制 / binlog 订阅
  ```
- **典型问题追问**：
  - "删缓存和更新 DB 哪个先做？" → 先删缓存再更新 DB（防并发读脏数据）
  - "如果删缓存失败怎么办？" → 重试队列 + 最终一致

## 🟢 B 级 — 方向对但有缺漏

### 1. MQ 消息丢失处理
- **你的回答**：提到 ACK 机制但缺少持久化
- **补充内容**：
  - 需要理解：生产者 ACK = all、broker 持久化、消费者手动 ACK
  - 常用命令：`acks=all`, `min.insync.replicas=2`
  - 需要考虑消费端的幂等设计

## 📝 沟通表达建议
- 回答结构不够清晰，建议使用 STAR/PREP 框架
- 加强对技术决策的 trade-off 表达能力

## 🎯 下次面试重点准备
1. Kafka 消息模型 + 选型对比（D 级盲区，优先级最高）
2. 分布式事务方案对比（C 级薄弱点）
3. 缓存一致性策略（B 级需完善）

## 行为面试评分标准（STAR 评估）

| STAR要素 | 评分点 |
|---------|--------|
| Situation | 是否清楚描述了背景和上下文 |
| Task | 是否明确了自身目标和责任 |
| Action | 是否具体说明了做了什么、为什么这么做 |
| Result | 是否有可量化的结果（数据、影响、反馈） |
| 反思 | 是否有总结学到了什么、会怎么改进 |

## 回答质量评估

### 结构评估
- **PREP 框架**：Point → Reason → Example → Point
- **STAR 框架**：Situation → Task → Action → Result
- 评估用户回答是否使用了结构化表达

### 反模式检测
| 反模式 | 示例 | 引导提示 |
|--------|------|---------|
| 规避问题 | "这个我记不清了" | 引导尝试根据原理推测 |
| 过度模糊 | "我们做了优化" | 追问具体做了什么、效果如何 |
| 甩锅他人 | "那是别人的问题" | 引导聚焦自己的角色和贡献 |
| 虚化数字 | "大幅提升了性能" | 追问具体提升百分比 |
| 面面俱到但无重点 | 一口气说三个项目 | 引导聚焦最有价值的项目 |

## 边界

- ✅ 基于简历项目出题、追问、实时提示、评分、学习报告
- ✅ 追问简历技术栈相关八股文（范围限定在简历技术栈范围内）

- ✅ 全流程面试模拟
- ✅ 中/英双语面试
- ❌ 不做简历修改（交给 `resume-builder`）
- ❌ 不做与简历无关的随机八股（不是通用题库）
- ❌ 不做算法题 / 手撕代码
- ❌ 不做具体薪资谈判指导（仅模拟场景）

## Common Mistakes

| 错误 | 修正 |
|------|------|
| 一次问多个问题 | 一次只问一个，等用户回答 |
| 答不上来直接给完整答案 | 先给提示，用户放弃才给框架 |
| 八股跑偏到简历没涉及的领域 | 严格限定在简历技术栈范围内 |
| 压力风格影响评分公正性 | 风格只影响语气，评分标准统一 |
| 死磕一个问题不放 | 追问最多 3-4 层，然后切换 |
| 忘记告知进度 | 切换项目时明确告知 |
| 忽略沟通表达评分 | 需评估回答结构完整性和清晰度 |
| 行为面试问成技术面试 | HR面严格使用 STAR 法，不追问技术细节 |
